# Workflow Rules

## 状态

- `TODO`: 尚未开始。
- `IN_PROGRESS`: 正在执行。
- `BLOCKED`: 遇到阻塞，需要输入、权限或外部变化。
- `REVIEW`: 已有产出，等待验收或下一步。
- `DONE`: 已完成并记录。
- `CANCELLED`: 已取消。

## 标准流程

1. 管理员创建任务票据。
2. 管理员指定唯一主负责人。
3. 主负责人把状态改为 `IN_PROGRESS`。
4. 主负责人执行任务并记录关键过程。
5. 如需其他 Claw 辅助，必须写清辅助任务和交付物。
6. 主负责人提交结果到 `REVIEW`。
7. 管理员按验收标准检查。
8. 通过后改为 `DONE`，并写入日志和记忆库。

## 防打架规则

- 一个任务只能有一个 `Owner`。
- 一个文件同一时间只能由一个 Claw 修改。
- 修改前先在任务票据写入 `Lock`。
- 跨 Claw 协作必须通过任务票据交接，不靠群聊口头记忆。
- GitHub 任务使用独立分支：
  - `agent/online/<task-id>`
  - `agent/desktop/<task-id>`
- 公开发布内容必须先进入 `REVIEW`。

## 交付标准

每个任务必须包含：

- 最终产物。
- 验证方式。
- 已知问题。
- 下一步建议。
- 是否需要用户确认。

## 管理员日报格式

```text
日期：

已完成：
- 任务编号：产出

进行中：
- 任务编号：负责人，当前进度

阻塞：
- 任务编号：阻塞原因，需要谁处理

待用户确认：
- 事项：为什么需要确认，建议选项

明日建议：
- 优先级 1
- 优先级 2
```
