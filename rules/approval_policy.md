# Approval Policy

目标：减少无意义确认，让智能体可以连续推进任务，同时保留高风险动作的人工确认。

## 无需确认，可以自动执行

- 读取公开网页和公开资料。
- 阅读用户指定仓库和当前任务相关文件。
- 创建草稿、提示词、方案、任务票据、日志。
- 在用户指定 GitHub 仓库创建新分支。
- 在新分支提交代码。
- 运行测试、构建、格式化、截图验证。
- 修改任务范围内明确指定的文件。
- 生成待发布文案、图片提示词、发布计划。
- 把任务状态从 `TODO` 改为 `IN_PROGRESS`、`REVIEW`、`DONE` 或 `BLOCKED`。

## 必须确认

- 删除文件、删除分支、删除仓库。
- 修改主分支、强推、重写 Git 历史。
- 公开发布社媒内容。
- 发送私信、评论、关注、取关。
- 使用付费 API、购买服务、消耗明显额度。
- 修改账号密码、密保、邮箱、手机号、支付方式。
- 上传或分享 token、cookie、私钥、身份证、银行卡等敏感信息。
- 修改系统级网络、安全、启动项设置。
- 执行可能导致断网、丢数据或账号风控的操作。

## 默认执行口令

用户发出以下话术时，执行者应连续推进到 `BLOCKED`、`REVIEW` 或 `DONE`，不要每一步询问：

```text
按预授权规则执行，允许连续推进。只有 approval_policy.md 中必须确认的事项才问我。
```

## 阻塞处理

如果遇到阻塞，执行者必须报告：

- 阻塞原因。
- 已尝试的办法。
- 需要谁处理。
- 最小确认问题。
- 可替代方案。
